### 0.1 展示 / 复习双状态（产品边界）

- 这是无需登录的内部工具，**两种状态只区分呈现内容，不承担访问控制**。仓库不部署上线，不把 clone 后的
  本地行为当作需要防护的安全边界。
- **展示（demo，默认）**只呈现中性技术说明；**复习（review）**额外展开 W3 的「仍在路上」开放问题与
  自我复盘。两种状态必须在页面上明确标识，不能靠使用者猜测。
- 内容状态落在 **URL hash**（`?mode=review`），不再存 `localStorage`：干净链接永远是展示状态，
  上一次复习不会悄悄残留进下一次演示。
- 开关**直接放在展板上**（不在登录页）：当前状态一眼可见、可随手切换；进入复习状态时给醒目横幅，
  提示内部 demo 前切回展示。
- **待复核结论的口径提示不受内容状态影响**，展示状态也照常显示，避免 demo 只呈现过度确定的版本。
- tab 统一写进 URL hash（`tab=`）；需要深链接定位复习位置的 W3 / W5 同时写入当前专题（`topic=`）。
  认证与 OAuth2 当前按顺序演示，不要求持久化内部步骤。

## 1. 目的

- 本周（W5，Node.js 底层）是核心保护周，但随时可能被面试等打断。
- 对策：**能学多少学多少，已验收的内容再用带 UI 的方式呈现**。展示可以帮助回看，但不计作复盘或掌握验收；partial 进度也不等于零。
- 载体：`week8-fullstack/src/frontend/`（AGENTS.md 白名单，AI 可维护）。

## 2. 边界（红线）

- ✅ AI 可做：把**本人已跑出的现象、数值、判断表**用 UI 呈现。
- ❌ AI 不做：事件循环 / 阻塞 / stream / worker 等**核心 demo 代码**（黑名单，援助上限 L2）。
- 分界：展示板画的是「本人产出的数据 + 本人推导的模型」，**不是**「输入任意代码→自动算执行顺序」的在线模拟器——后者等于替本人完成理解 demo。

## 3. 设计原则

1. **少字多象**：每主题以一个可视化的可观察现象为主角，文字降为旁注。
2. **真实数据驱动**：画的是本人实验测到的真实数值，不是装饰图形。
3. **粗颗粒**：每周 demo 仅约 10 分钟，做「几个能一眼看懂的大块」，不做细碎分步。

### 3.1 两个交付轴（2026-07-22 已拍板）

- **可视化复习轴**：按知识点组织，覆盖已经完成并验收的学习成果；目标是帮助本人回看、比较和重新推导，不受 demo 时长限制。
- **Demo 展示轴**：从复习板中挑少数知识点精讲，单独编写讲稿；不要求页面结构等于演讲顺序，也不要求展示全部学习内容。
- 两者解耦：复习板追求知识结构完整，讲稿追求有限时间内的重点与叙事。
- 页面弱化 D1 / D2 等日程标签，统一改用「知识点 1 / 2…」；未完成主题不提前占据主视图。

### 3.2 统一约定：知识点切分（2026-07-23 定，适用于所有复习板）

所有复习板（认证、OAuth2、数据库聚合、运行时，以及后续 W1 / Stream 等）都遵守下面这套切分口径：

1. **切分主轴是「知识点递进 / 相关」，不是「哪天学了什么」**：复习按概念组织，日程只留在笔记来源（`source`）里。
2. **每个板 = 顶部知识点导航 + 板头一句话递进说明**：相关知识点相邻放置；有依赖关系的排在被依赖者之后
   （如 `$lookup 外键索引` 依赖 `explain 复合索引`，排在其后）。
3. **只纳入已验收的知识点**。已验收但**非本机实测**的「链路理解 / 判断模型」（如「I/O 慢」）也可作为知识点纳入，
   但必须在结论页脚与来源行标清「非实测」，与实测知识点区分。
4. **待复核 / 未验证的结论**进「仍在路上 / 开放问题」面板，或用「结论口径」提示限定为当前实验证据、不推广成
   通用判据；**口径提示对所有观众可见**（展示 / 复习都显示）。
5. **综合类落点**（如「三类慢判断表」）作分诊 / 索引入口，**链接回对应知识点**，不重复承载深挖内容。
6. 各板当前知识点顺序（记录，便于回看与回填）：
   - **认证与授权**：注册（密码存储边界）→ 登录（凭据验证）→ JWT（身份恢复）→ RBAC（401/403 授权）。
   - **数据库聚合（W3）**：先写对（分层归位 → 自然月边界）→ 再调快（explain 复合索引 → `$lookup` 外键索引）。
   - **运行时（W5）**：事件循环调度边界 → CPU 阻塞 → 线程池排队 → 外部 I/O 等待（判断模型，非实测）。

### 3.3 收尾纪律：掌握证据 vs 对外表达（2026-07-23 定，每周 / 全栈收尾通用）

- **两层**：**掌握证据**（核心 demo、脱稿串讲、复盘、债务重建）由本人产出、可粗糙，是闸门；
  **展示派生物**（展板的展示 / 复习状态、汇报 demo、讲稿）可由 AI 打磨，但只能处于掌握证据的下游——
  只搬运 / 可视化已验证的结论，不生成推导。前端的 `review` 只是内容状态，不是一次掌握复盘，也不是验收通过标记。
- **先自证，后润色**：掌握证据必须先于任何 AI 润色产出。若「把它做漂亮」的请求会让 AI 写出核心推导，
  AI 当场标出并停在「渲染已有结论」那一层，不替本人完成理解。
- **来源闸门**：已验证结论必须能回指本人笔记、原始实测或重建记录；待验证内容只能以「开放问题 / 结论口径」呈现，
  不能由 AI 补成确定结论，也不能因进入展板而改变验收状态。
- **一个产物不同时承担两个角色**：证明用的产物自制，展示用的产物派生，不合并成一个（否则分不清是本人懂了还是 AI 写了）。
- **碎片时间不是排期义务**：需要专注的掌握证据只放完整时间块；碎片时段可以由 AI 独立处理展示层，
  但不因此要求本人当时审核，周末默认不安排学习或展示 review。

## 5. 各主题视觉呈现（定调，非最终）

W5 运行时板现有 **4 个知识点**，按运行时判断递进（调度边界 → CPU 阻塞 → 线程池排队 → 外部 I/O 等待）；
底部「三类慢判断表」把后三者收成一张分诊表，每张卡片可跳回对应知识点：

- **知识点 1 · 事件循环调度边界**（已完成，实测）：调用栈、nextTick、microtask、libuv 阶段四层模型呈现推导顺序；顶层与 I/O callback 场景作为对照。
- **知识点 2 · 同步 CPU 阻塞**（已完成，实测）：真实时间线对照 `20ms→late 0` 与 `2000ms→late 1904`，标出 CPU 占用、timer 到期和 callback 执行位置。
- **知识点 3 · 线程池排队与 UV_THREADPOOL_SIZE**（已完成，实测）：pbkdf2×8 在 SIZE=4/8 下的分批 vs 聚批，worker/队列机制示意 + 排队回放。
- **知识点 4 · 外部 I/O 等待**（已完成，**链路理解 / 判断模型，非实测**）：分诊推理（观测→排除/指向→工作假设→证据边界）、等待远端 2s 内 OS/libuv/Node/V8 四层职责泳道、poll 等待 vs 同步 while 对照、fd 释义。来源口径已标「非本机实测」。
- **stream / 背压**（未开始）：producer/consumer 缓冲条动画，高水位触发 drain 变色；旁边 `readFile` vs stream 内存占用对比条。
- **错误边界 / worker**（未开始）：错误流向图、主线程阻塞 vs worker offload 响应能力对照——学到后回填。
