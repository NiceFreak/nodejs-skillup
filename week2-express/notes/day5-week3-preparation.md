# Day 5 · Week 3 预习 · Mongoose 进阶与查询优化

> Week 3 的主题是 **Mongoose 进阶与查询优化**，核心会落在两件事上：**聚合管道（aggregation pipeline）** 和 **查询性能**。今天不写代码、不背语法，只提前建立几个心智模型：聚合管道解决什么问题、它和 Express 中间件有什么相似之处、几个核心 stage 分别对应什么概念，以及为什么 `$match` 的位置会影响性能。

---

## 1. Week 3 主线

Week 3 要从「能 CRUD」继续往前走一步：不只是把文档查出来，而是学会在数据库侧完成更复杂的数据加工。

两条主线：

| 主线 | 要解决的问题 |
|---|---|
| **聚合管道** | 数据不只是原样返回，而是要筛选、分组、统计、关联、重塑后再返回 |
| **查询性能** | 查询能不能用上索引，数据量变大后会不会退化成全表扫描 |

这两条线其实是连在一起的：聚合管道负责「怎么加工数据」，查询性能负责「这个加工过程是不是高效」。

---

## 2. 聚合管道解决的是什么问题

`find()` 做的是「筛选文档，然后原样返回」。它适合这种需求：

- 查所有用户
- 按 `_id` 查一个用户
- 查某个条件下的订单列表

但很多真实需求不是单纯筛选，而是要**先加工数据再返回结果**，比如：

- 按省份分组，统计每个省有多少用户
- 找出订单里每个商品的销售总额
- 把 users 和它们的订单关联起来，一次查出来

这些需求的共同点是：返回结果已经不是原始文档本身，而是经过计算、分组、关联或重塑后的新形状。

**聚合管道要解决的就是这类问题：把一批文档经过一连串加工步骤，变成最终想要的结果。**

---

## 3. 心智模型：Express 中间件管道，换了一个处理对象

这是今天最重要的预热点：**聚合管道和 Express 中间件管道是同一种「管道式处理」思想，只是处理对象不同。**

| 场景 | 流过管道的东西 | 每一环做什么 |
|---|---|---|
| Express 中间件 | HTTP 请求 / 响应 | 校验、解析、鉴权、错误处理 |
| MongoDB 聚合管道 | 一批文档 | 筛选、分组、排序、改形状、关联集合 |

Express 中间件是请求依次经过 `A -> B -> C`，每一层对请求做一点处理，再交给下一层。

聚合管道是一批文档依次经过 `$match -> $group -> $sort -> $project`，每一个 stage 对数据做一点处理，再把结果交给下一个 stage。

```js
Model.aggregate([
  { $match: { ... } }, // 第一步：先筛选
  { $group: { ... } }, // 第二步：把筛出来的数据分组统计
  { $sort: { ... } },  // 第三步：把统计结果排序
])
```

关键点：**前一个 stage 的输出，就是下一个 stage 的输入。**

所以 Week 3 学聚合时，可以直接复用 Day 1 建立的直觉：数据会依次流过一串处理站，每个处理站都只负责其中一步。

---

## 4. 核心 stage 先认脸

今天不用记具体语法，只要知道这些 stage 大概负责什么，以及它们对应 SQL 里的哪个概念。

| Stage | 作用 | SQL 类比 | 先记住的直觉 |
|---|---|---|---|
| `$match` | 筛选文档 | `WHERE` | 尽量放前面，先把不需要的数据过滤掉 |
| `$group` | 分组统计 | `GROUP BY` | 聚合里最需要花时间理解的 stage |
| `$sort` | 排序 | `ORDER BY` | 对结果排序 |
| `$project` | 重塑输出字段 | `SELECT` 选字段 | 控制最终返回哪些字段，或计算新字段 |
| `$lookup` | 关联另一个集合 | `JOIN` | MongoDB 里的关联查询 |

`$group` 会是 Week 3 的重点，因为它不只是「查」，而是会改变结果的结构：原来是一条条文档，分组后会变成一组组统计结果。

`$lookup` 也很关键，因为它连接回 Week 1 的「嵌入 vs 引用」建模问题。

---

## 5. `$lookup` 和 Week 1 建模的连接

Week 1 讨论过「嵌入 vs 引用」：

- **嵌入**：相关数据直接放在同一个文档里，读取简单，但文档可能变大、重复数据可能变多。
- **引用**：数据拆到不同集合里，用 id 建立关系，结构更独立，但读取时通常要再关联。

引用型建模的代价是：数据分散在两个集合里。没有聚合时，常见做法是手动查两次，再在应用层自己拼起来。

`$lookup` 的作用就是在数据库侧完成这件事：**在一次聚合查询里，把两个集合按照某个字段关联起来。**

所以，Week 1 里那些「这里更适合用引用」的场景，到了 Week 3 会补上对应的取数方式。

一句话连接起来：

> 建模时选择「引用」，查询时就要知道怎么「关联」；`$lookup` 是 MongoDB 里处理关联查询的重要工具。

## 6. 查询性能预热：索引依然重要，但 stage 顺序会影响它

Week 1 学过：

- `explain()`
- `COLLSCAN` vs `IXSCAN`
- 最左前缀
- 查询条件能不能命中索引

这些知识到聚合管道里依然有用，但要多注意一个新问题：**索引通常更容易在管道早期 stage 发挥作用。**

尤其是 `$match`：

- 如果 `$match` 放在最前面，它很像普通 `find()` 的筛选条件，有机会使用索引。
- 如果先 `$group`、`$project` 或其他 stage 把数据形状改变了，再 `$match`，这时数据已经不是原始集合里的原样文档，索引可能就用不上了。

所以 `$match` 尽量前置有两层意义：

1. **减少后续 stage 要处理的数据量**：先过滤掉不需要的文档。
2. **尽量保留使用索引的机会**：越靠近原始集合，越可能命中索引。

Week 3 正式写聚合时，可以用 `explain()` 验证这个判断，而不是只凭感觉认为「看起来应该更快」。
