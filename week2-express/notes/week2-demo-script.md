## 2. 架构讲解（3 分钟，指着目录/代码讲，不用念代码）

**做什么：** 打开 `week2-express/src/` 目录，照着这张图指一指就行：

```
src/
├── app.js                 # 挂中间件、路由、错误中间件、优雅关闭
├── config/db.js           # connectDB / disconnectDB
├── middlewares/           # 入口把关：ObjectId 格式、body 是否存在
├── models/users.js        # Schema + 校验(required/unique/match)
├── routes/users.js        # URL → 中间件 → controller
├── controller/users.js    # 处理 req/res，调用 service
├── services/users.js      # 业务规则(目前装的是字段白名单)
├── repositories/users.js  # 数据访问 + 错误翻译
├── errors/userErrors.js   # 领域错误类
└── utils/validators.js    # 纯校验函数
```

**讲三件事，带这几个关键词：**

**① 分层 —— `四层` `各层职责单一` `req/res 只在 controller`**
> "整个项目分四层：请求先到 **route**，匹配走哪个处理函数；到 **controller**，负责从 `req` 里取参数、调用 service、把结果包成响应返回；**service** 放业务规则；**repository** 才真正操作数据库。**`req`/`res` 只在 controller 里出现**，这条规矩从第一天到现在没破过，其他层不碰。"

**② 校验放哪 —— `跟业务没关系的` `业务规则` `中间件` `service`**
> "校验我分了两种。一种**跟业务没关系**，比如这个 id 格式对不对、请求有没有带 body——这种在最前面的 **中间件** 那层就直接拦掉了，压根进不到业务逻辑里。另一种是**业务规则**，比如'一个用户哪些字段能被改'，这个我放在了 **service** 里——为啥这么分等下 Q&A 细说。"

**③ 报错走两条路 —— `404` `409` `冒泡` `两条路互斥`**
> "有一个我觉得挺值得讲的点：**查不到东西返回 404**，跟**数据冲突返回 409**，走的是完全不同的两条路。404 是 controller 自己判断出来的，就是很普通的一次函数返回；409 是数据库那边真出错了，这个错误会一路**冒泡**，冒到最上面一个专门处理错误的中间件去统一处理——两条路**互斥**，一次请求只会走其中一条。"

## 3. 黄金路径实操（Postman，5 分钟）

**做什么：** Postman 打开文件夹 **"一、黄金路径 · 完整生命周期演示"**，从上到下点。请求之间会自动传 `demoUserId`，不用手动复制粘贴 id。

| # | 点这个请求 | 关键词 | 大白话怎么带 |
|---|---|---|---|
| 1 | List users（创建前） | **先看一眼现状** | "先点一下列表，看看现在库里有啥，等下好对比" |
| 2 | Create user | **201** · **内嵌的地址字段** · **落库** | "点一下，201，返回新建的用户，带了个自动生成的 `_id`，还有一个内嵌的地址字段——" → 切 Compass 刷新 → "看一下，数据确实写进去了" |
| 3 | List users（创建后） | **确认写入生效** | "再点一下列表，新用户已经在里面了" |
| 4 | Get user by id | **按 id 查** | "按 id 查一下，内容跟刚创建的一样" |
| 5 | Update（PATCH） | **局部更新** · **PATCH 不是 PUT** | "改一下名字和年龄，200。这里用的是 **PATCH**，只传想改的字段，别的不动——为什么不用 PUT，后面 Q&A 会讲" |
| 6 | Get 确认更新持久化 | **改动是真生效的** | "再查一次，确认改动确实写进去了，不是接口返回了 200 但数据其实没变" |
| 7 | Delete | **200** · **返回一句话而不是对象** | "删掉，200，返回一句'删除成功'——没返回被删的那个对象，因为东西都没了，给完整数据意义不大" |
| 8 | Get 确认已删除 | **404 · controller 自己判断的** | "再查一次，**404**——这个 404 就是刚才说的，controller 自己判断'查不到就返回 404'，没走报错那条路" |

## 4. 边界与错误场景（5 分钟，挑着讲，时间紧就只讲加粗的两条）

**做什么：** Postman 里 **"二/三/四"** 三个边界文件夹，挑着点：

- **400（格式不对/校验不过）**
  - **`Create user - invalid email format`** → 400。**大白话：** "邮箱格式不对，直接拒绝——这条其实是我自己手写测试清单的时候挖出来的一个真漏洞，当时是能创建成功的，后来在 Model 上补了个正则校验才堵上"
  - `Update user - empty body (no valid fields)` → 400。**大白话：** "传个空 body 改用户，没有一个字段能改，这种也拦下来，不让它返回 200 但实际什么都没改"
- **404（东西不存在）**
  - `Update user - valid format, no matching user` → 404。**大白话：** "id 格式是对的，但查无此人，跟前面那套判断逻辑一样"
- **409（邮箱冲突）**
  - **`Create user - duplicate email`** → 409。**大白话，顺带把整条链讲完整：** "邮箱重复了，409。这条错误怎么来的——**MongoDB 先报一个重复键的错误码，repository 这层捕获这个错误码，把它转换成一个'邮箱冲突'的业务错误对象并抛出，这个错误顺着调用链一路冒泡，冒到最上面统一处理报错的错误中间件，那边判断错误类型、赋 409 返回**。中间 controller 完全没有 catch 这个错误，也没做任何转发，是错误自己冒泡上去的。"
  - `Update user - duplicate email` → 409。**大白话：** "改邮箱改成重复的，一样是 409——说明这条报错的路子不是 Create 专属的，Update 直接复用了同一套逻辑"

## 5. 优雅关闭演示（1 分钟，终端操作）

**做什么：** 切到跑 server 的终端，按 `Ctrl+C`。

**带这几个词：** **不是直接杀进程** **先停新请求** **再断库** **最后退出**

**大白话怎么说：**

> "最后演示一个细节——**优雅关闭**。如果直接把进程杀掉，正好有请求在处理中的话，会被中断，数据库连接也来不及正常关闭。"

按下去，终端应该依次跳出：

```
Received SIGINT. Shutting down gracefully...
Disconnected from MongoDB
SIGINT Server closed
```

> "这个顺序是——**先停止接收新请求、把正在处理的请求跑完，再断开数据库，最后才退出进程**。顺序不能反，反了的话，正在处理的请求这时候想查库，库已经断了，直接就报错了。"

## 6. 设计取舍 Q&A 预案（被问到直接照这个答）

**Q：为什么没写 PUT，只有 PATCH？**
> **关键词：局部更新 / 没有真实场景**
> "PATCH 就够用了，因为我这几个改动场景都是'改几个字段'，没有一次是要把整条记录整个换掉。PUT 那种'整体替换'的场景现在没遇到过，等真有需求了再补——这是我自己权衡过的，不是漏写了。"

**Q：为什么允许地址 `addresses` 传空数组？**
> **关键词：新用户没填地址很正常**
> "刚注册的用户可能还没填收货地址，这种情况应该让它能创建成功，不该在这一步就卡住人。这是我主动做的产品判断，不是 bug。"

**Q：报错返回的格式是 `{ error: "..." }`，为啥不搞得更规范一点，比如带个错误码？**
> **关键词：记了 TODO / 改动面大不能顺手改**
> "这个我记在 TODO 里了，还没做。因为改动范围不小——不只是改错误那部分，所有接口成功返回的格式也得跟着统一，怕改一半新旧格式混着用出问题，所以先放着，等专门空出时间来做。"

**Q：为什么'哪些字段能改'这个校验放在 service，不放中间件里？**
> **关键词：换个业务还成不成立**
> "我自己定了个标准：这件事说的是'HTTP 请求长什么样'，还是'这个业务允许发生什么'。像 id 格式对不对，换个什么业务都一样，这种放中间件；但'用户允许改哪些字段'这种，换一个业务（比如库存系统）规则完全不一样，这就是业务规则，该归 service。"

**Q：数据库报错的原始信息直接返回给前端，合适吗？**
> **关键词：调试方便 / 已知要改**
> "现在是直接透传的，确实比较啰嗦，是留着方便调试用的。生产环境一般不会把内部报错原样返回给客户端，这个也记在 TODO 里了，demo 阶段先不动。"
