**问题在这句:"再返回到错误中间件中处理,最后 controller 抛出"——顺序和主体都反了。**

正确的流向是:

```
repository 抛出领域错误(EmailConflictError / UserValidationError)
  → 这个错误顺着 async 链一路"冒泡"网上传
  → controller 完全没有 catch 它(Express 5 帮你自动捕获 async 函数里的 throw)
  → 错误跳过 controller、跳过所有普通中间件
  → 直接被 app.js 那个四参数的错误中间件接住
  → 错误中间件里判断 instanceof、赋 statusCode、res.status().json()
```

**controller 不"抛出"任何东西——它甚至不知道这个错误存在。** 错误是从 repository 直接冒泡到错误中间件的,controller 只是这条冒泡路径**经过**的一层,它没有拦截、没有处理、也没有转发的动作。这正是 Express 5 那个特性的意义:你不需要在 controller 里写 try/catch 把错误"捡起来再扔给中间件",错误自己会顺着调用栈跳过所有中间层,直达错误中间件。

**一个更精确的类比帮你记**:想象错误是个警报信号——它不是被 controller"接住再转手抛出去"的,是**穿透**了 controller、穿透了没有 catch 它的每一层,直接被专门监听警报的那个装置(错误中间件)截获。中间没人"处理"它、也没人"抛"它——**只有 repository 这一处主动 throw**,之后全靠"没人 catch 就自动往外冒泡"这个机制,一路冒到底。

**回到你那句话,改法很简单:把"controller 抛出"删掉**。正确版本应该是:

> 400/409 是 repository 中主动抛出的,对应 schema 校验失败和 Mongo 的 11000。我们在 repository 把这类底层错误翻译成业务错误(领域错误),然后**这个错误直接冒泡到错误中间件**(controller 没有拦截它),由错误中间件统一判断类型、赋状态码、返回响应。

**但你的错误处理(400/409/500)走的是另一条路径,这条路径大多数入门教程根本没画,或者画得很简化。**

问题就出在这——**很多网上的分层架构图,画的是"理想的、没有异常的正常流程"**,它默认所有响应都走 controller,是因为那种教程通常还没引入"全局错误中间件"这个机制,或者把错误处理简化掉了,只画了 happy path。这类图不是错,是**不完整**——它没画出"当 repository 抛错时,响应从哪里发出去"这条岔路。

**在你的实际项目里,有两个"发响应"的出口,不是一个:**

1. **正常路径出口:controller。** `res.status(200/201/404).json(...)` —— 你手动写的,你在 controller 里判断该返回什么、调用 res。
2. **错误路径出口:错误中间件(app.js 里那个四参数函数)。** 它同样调用了 `res.status(...).json(...)` —— 但这段 `res` 调用**不在 controller 文件里**,在 app.js 的错误中间件里。这也是"发响应"的动作,只是发生的位置不是 controller,而是错误中间件自己。

**两个出口的共同点是:都只碰一次 res、都是整条链上唯一"真正对外发响应"的地方。区别只是——正常情况下这个角色是 controller 演的,出错情况下这个角色换成了错误中间件来演。** 图如果要画完整,应该在 controller 旁边再画一条"抛错 → 错误中间件 → res"的分支路径,而不是让所有箭头都汇回 controller。

**一句话消掉这个矛盾:** "response 从 controller 出去"这句话,准确说应该是"**response 从『离用户最近、且真正处理这次请求结果的那个函数』出去**"——大多数时候那个函数是 controller,但**一旦发生了未被 controller 自己 catch 的错误,那个函数就换成了错误中间件**。图没画错,只是没画全,画的是简化版的 happy-path 世界。

- **错误响应**:repository throw → 冒泡(中间没人碰、没人处理)→ 错误中间件是另一个出口,它自己 `res.status().json()`。

这两条路径**互斥**——一次请求要么走 controller 这条(正常返回,包括你手动判断的 404),要么走错误中间件这条(抛错触发,400/409/500),不会同时发生。

- 还是 **传了 `addresses: []`**(有这个 key,但是空数组)

这两种在 Mongoose 眼里,最终结果是一样的——都会成功创建,而且**存进数据库的 `addresses` 都是 `[]`**。原因分两层:

**第一层(基础的,你大概率已经知道):`addresses` 这个字段本身没有 `required: true`。** 你 Schema 里只给数组**里面每个元素的字段**(recipient、phone 等)加了 required,没给 `addresses` 这个字段整体加 required。所以"要不要传这个字段"这件事,Mongoose 不做任何强制——不传就是合法的。

**第二层(这才是回答你问题的关键,也是刚才你自己验证过的那个现象):Mongoose 对"数组类型字段"有个特殊的默认行为——即使你完全不传,它也会自动把这个字段初始化成空数组 `[]`,而不是让它变成 `undefined` 或者压根不存在。**

这是 Mongoose(不是 MongoDB 原生,是 Mongoose 这层 ODM 的设计选择)对 Array 类型的默认值处理:**Schema 里声明了 `[...]` 类型的字段,不管你传不传,Mongoose 都会给它一个默认的空数组**,除非你显式设置了别的 default。这跟 String/Number 这些类型不同——那些类型不传的话,不写就真的是"没有这个字段"(除非你设了 default)。数组类型比较特殊,它的"空状态"天然就是 `[]`,不是"没有"。

**所以完整的因果链是:**

```
addresses 没有 required 校验
  → Mongoose 不强制要求你传这个字段
    → 你不传 / 传空数组,都合法,能通过创建
      → 存进数据库时,Mongoose 自动把它落地成 []（数组类型的默认行为）
```

**这跟你已经确认过的另一条规则是完全一致的、不矛盾的**:`addresses` 数组**里面的元素**(一旦真的传了某个 address 对象进去)才会触发 required 校验——那是你刚才实测过的("缺 phone/city 会 400")。**"要不要传这个数组"和"数组里的元素合不合格"是两套独立的校验层级**,前者你没设限制,后者你设了。

**记成一句话:required 只管"这个字段存在还是不存在",不管"数组里有几个元素、内容对不对"。数组本身的"最少几个元素"是另一种校验规则(叫数组长度校验),Mongoose 的 `required` 不会自动帮你管这个——这也是你之前判断"想让空数组被拒绝需要额外加校验"完全正确的原因所在。**

## Delete:`DELETE /users/:id`

**你要自己想清楚的两个点:**

1. **删除一个不存在的 id,该返回什么?** —— 这个你已经有现成的模型了:`findByIdAndDelete` 查不到时返回什么(null,还是别的)?这个 null 该在哪一层被判断、翻译成什么状态码?**这跟你今天刚刚绕晕又理清的那条"404 是谁在主动判断"完全是同一套逻辑**,直接套用即可。
2. **删除成功,返回什么?** 这是 Delete 特有的新判断题——两种常见做法:
   - **`204 No Content`**:成功了,但没有 body 可返回(资源已经不存在了,没什么可给的)
   - **`200 OK` + 返回被删除的那个对象**:让客户端知道"删的是这个"
   
   两种都是业界常见做法,没有绝对对错。你选一个,但要能说出**为什么选它**——这是个设计判断,不是背答案。

写完测三条:合法且存在的 id → 你选的成功状态码;合法但不存在的 id → 404;格式非法的 id → 400。
