**C1:关联无索引字段的对照实验,怎么做**

核心思路和你 Day1 的"加索引前后对比"**完全一样**,只是从"`$match` 筛选"换到"`$lookup` 关联"。三步:

**第一步:造一个"关联无索引字段"的场景。**
你现在的 `$lookup` 关联 user 的 `_id`(有索引,所以快)。要看慢的,得关联一个**没索引的字段**。最简单的做法:

给 order 加一个查询,`$lookup` 关联 user 的 **`name`** 字段(name 没索引)。构造一个这样的管道(纯实验,不用接进正式代码):
```js
db.orders.aggregate([
  { $lookup: {
      from: "users",
      localField: "userId",      // order 的 userId
      foreignField: "name",      // ← 故意关联 user 的 name(没索引!)
      as: "userInfo"
  }}
]).explain("executionStats")
```
（这个关联在业务上没意义——userId 对 name 根本匹配不上——但**实验的目的不是拿到正确结果,是看执行计划**。你只看它怎么访问 users 集合。)

**第二步:看 explain,应该看到全表扫描。**
跑上面这个,看 `$lookup` 阶段的那两个关键字段(你今天刚学会看的):
- `collectionScans`:这次应该 **> 0**(不再是0)——因为 name 没索引,只能扫全表
- `indexesUsed`:应该是**空的**(没索引可用)

对比你今天关联 `_id` 的结果(`collectionScans: 0`、用了 `_id_`),差异一目了然:**关联主键走索引(scans=0),关联无索引字段全表扫描(scans>0)。**

**第三步:给 name 建索引,再跑,看它变快。**
```js
db.users.createIndex({ name: 1 })
```
然后再跑一遍同样的 explain。这次 `collectionScans` 应该回到 **0**,`indexesUsed` 里出现 `name_1`。**加索引前后的对照就出来了**——和你 Day1 那张 COLLSCAN→IXSCAN 的表同一个套路,只是这次是关联维度。

**这个实验的结论(值得进优化笔记):`$lookup` 的性能取决于 foreignField 有没有索引。关联主键或有索引的字段 → 走索引、快;关联无索引字段 → 全表扫描、慢;给关联字段建索引能消除全表扫描。**

**做完记得清理**:实验建的 `name_1` 索引可以删掉(`db.users.dropIndex("name_1")`),别留在库里。

**今天状态不好的话,这个实验可以不做**——你已经理解了原理(关联字段有无索引决定快慢),做实验只是"眼见为实"。理解了就能写进复盘的"没吃透→其实已经懂原理,只差亲手验证"。

**C2:Decimal128 转换放哪层,为什么还"没吃透"**

你现在放在 **service 出口**(`Number(totalSpending.toString())`)。这能跑、也合理。"没吃透"是因为存在一个**更规范的方案**,你还没接触,所以心里没底"我这样是不是最优"。

**你现在的做法:** service 从 repository 拿到原始数据(含 Decimal128),在返回前转成 number。转换逻辑散落在每个 service 方法里(客户报表转一次、月度报表转一次,代码重复)。

**更规范的方案:DTO / 序列化层。**

DTO(Data Transfer Object,数据传输对象)是一个**专门定义"对外返回长什么样"的层**。思路是:不在 service 里零散地转,而是有一个专门的地方,负责把"内部数据模型"转换成"对外响应格式"——Decimal128→number、`_id`→userId、剔除 `__v` 这些,**全在这一层统一做**。

**为什么它更好(也是你"没吃透"的点):**
- **消除重复**:现在你每个 service 方法都写一遍 Decimal128 转换。DTO 层集中处理,写一次
- **职责更纯**:service 专注业务逻辑,"怎么序列化成对外格式"交给 DTO 层。你现在是让 service 兼了"表示层转换"这个活,严格说不够纯
- **对外契约清晰**:DTO 明确定义了"这个接口返回哪些字段、什么类型",相当于 API 的"输出规格"

**为什么现在没做、也不该现在做:**
- 你现在只有2-3个报表,重复不严重,放 service 够用
- DTO 是"接口多了、重复明显了"才值得引入的抽象——**过早引入是过度设计**
- 它更适合 Week8 整合时,或者接口规模上来后再重构

**所以"没吃透"的准确表述**(可以这么写进复盘):"Decimal128 转换目前放在 service 出口,能用但有重复;了解到更规范的做法是引入 DTO/序列化层统一处理对外格式转换,但当前接口少、暂不需要,留待规模变大或 Week8 整合时再考虑。"——**这样写,显示你不只是'不知道',而是'知道有更好的、也知道现在为什么不用',这才是有深度的复盘。**

你把对照实验完整做出来了,而且抓到了核心——`collectionScans` 和 `indexesUsed` 在建索引前后确实变了。这份数据是完美的对照,我帮你把它读成一张清晰的结论表。

**你实验里那个"关联 name"的 `$lookup` 阶段,建索引前 vs 建索引后:**

| 指标 | 建 name 索引**前** | 建 name 索引**后** |
|---|---|---|
| `collectionScans` | **3**(全表扫3次) | **0** |
| `indexesUsed` | **[]**(空,没索引可用) | **["name_1"]** |
| `totalDocsExamined` | **15**(扫了整个users集合×多次) | **0** |
| `executionTimeMillisEstimate` | 12ms | 3ms |

**这就是查询优化最有说服力的一张对照,你亲手做出来了:**

- **建索引前**:关联 `name`(没索引)→ `collectionScans: 3`、`indexesUsed: []`、扫了 15 个文档 → **全表扫描,慢(12ms)**
- **建索引后**:`db.users.createIndex({ name: 1 })` → 同样的查询变成 `collectionScans: 0`、`indexesUsed: ["name_1"]`、`totalDocsExamined: 0` → **走索引,快(3ms)**

**对比你的主键关联(`_id`)那个 `$lookup`**:一直是 `collectionScans: 0`、`indexesUsed: ["_id_"]`——因为主键自带索引,天生高效。

**所以完整结论(直接进你的优化笔记):**

> `$lookup` 的性能取决于 `foreignField` 有没有索引:
> - 关联主键 `_id`(自带索引)或已建索引的字段 → 走索引(`IndexedLoopJoin`)、`collectionScans: 0`、快
> - 关联无索引字段(如 name)→ 全表扫描、`collectionScans > 0`、`indexesUsed` 为空、慢
> - 给关联字段建索引后,同一查询从全表扫描变为走索引(`collectionScans: 3→0`,`indexesUsed: []→["name_1"]`)
> 判断 `$lookup` 性能只需看两个字段:`collectionScans`(应为0)和 `indexesUsed`(应非空)。

**这和你 Day1 的 `$match` 优化(COLLSCAN→IXSCAN)是同一个道理的两个应用**:`$match` 靠索引避免扫全表,`$lookup` 靠 foreignField 的索引避免扫关联集合。**"该查的字段建索引"是贯穿两者的统一原则。**
