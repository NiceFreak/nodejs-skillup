# 存量项目生存指南 & 「我脱节了吗」校准

> 沉淀自 2026-07-16/17 的两轮讨论。背景：多年维护大型存量项目（React 16 及更低、
> Next 14/15、Express、CJS/ESM 混用、版本不做大更新），担心与新生态脱节，
> 且不确定这种担心是否真实（面试信号噪声大）。
> 结论先行：**不是脱节，是深而窄——窄的部分是一张可枚举的短清单。**
> Hooks 缺口单独见 [`react-hooks-interview-map.md`](./react-hooks-interview-map.md)。
> 白名单资产，AI 整理维护。

## 一、技术画像校准：你的栈就是生产世界的中位数

把「脱节感」拆到具体技术上逐项核对：

| 你的栈 | 实际状况 |
|---|---|
| Express | Node 世界使用量最大的框架；Express 5 在 Express 4 统治近十年后才发布。本仓库 skillup 也在用它 |
| CJS/ESM 混用 | 全行业日常，Node 官方自己的迁移都没走完（`require(esm)` 近期才支持）。没有混用问题的项目只有一种：刚创建的 |
| React 16 存量 | 企业生产版本显著滞后最新版是常态；React 19 有迁移成本，大项目不动是理性决策 |
| Next 14/15 | **现役主流**，不是旧技术。RSC / App Router 经验比「只写 Vite SPA」更接近 2026 前沿话题 |
| 版本不做大更新 | 多数情况是理性的业务决策（成本收益计算），不是技术怠惰——亲历过这个决策逻辑本身是经验 |

## 二、两个世界

「与世界脱节」的问题在于「世界」指哪个：

```text
广播世界：掘金 / Twitter / 技术大会，以「本周新发布」为节奏，音量大
生产世界：代码真正运行、钱真正流动的地方，以「年」为迁移节奏，占比大
```

焦虑通常来自拿广播世界的更新频率当自己的考核标准。广播世界的音量和它在生产中的
占比严重不成比例；你在生产世界里深度接轨了多年。

## 三、怎么校准「我是否落后」——用可证伪的信号，不用感觉

**噪声大的信号（别用来校准自我评估）：**

- 面试问题的难度与方向：反映的是面试官的水平和心情，不是市场要求
  （实例：5 年经验被问「JS 有哪几种基本类型」）。
- JD 文本：大量 JD 是模板拼贴，信息量低（2026-07-17 讨论结论：不做 JD 抽样统计）。

**可用的信号（两个就够）：**

1. **Gap 清单燃尽**：把「我不会的」写成有限列表。真脱节的特征是列表写不完；
   「深而窄」的特征是几行写完（当前实例：hooks 深度、React 18/19 新特性、
   现代工具链 Vite/Query/Zustand——三行，以周计）。
2. **亲手碰新栈**：读一遍现代栈代码、跑一个小项目。实测：本仓库前端（Vite + TS +
   hooks）能无障碍读懂每一行——**primitives 没变**（组件/状态/副作用/请求/构建），
   变的只是表层 API，而表层 API 的重获取速度以天计（JWT、RBAC、聚合管道均为实例）。

## 四、存量项目生存纪律（直面现实）

进烂项目/接手存量代码的前几周，动作是固定的：

1. **先跑起来，再读，最后才改。** 本地跑通 + 能打断点，胜过读三天代码；
   跑不起来本身就是第一张地图（缺什么环境、哪份文档是假的）。
2. **顺着一条真实请求走到底（竖切）。** 挑黄金路径（电商 = 下单），从路由入口跟到
   数据库再回来。烂项目没有分层，但请求路径永远存在；心里的「理想形状」
   （route → controller → service → repository）是阅读透镜——能看出 2000 行的
   controller 里哪段其实是 service。
3. **Chesterton 栅栏**：不明白为什么存在的代码，先假设它有理由。`git log -p` 和
   blame 是历史书，那行诡异的 `if` 后面往往是一次线上事故。
4. **改动前锁行为**：没有测试的代码，先给要动的那块写特征测试
   （characterization test——不管对错，先把现状断言下来），再动手。
5. **绝不上来就重构。** 用小胜利换信任；童子军规则只作用于碰到的文件；
   换旧模块用绞杀者模式（新功能走新路径，旧代码围而不攻）。
6. **改存量接口前盘点所有消费方。** 测试也是消费方（本仓库 D3 的 CI 红灯复盘）。

## 五、快速适应业务（代码之外的一半）

1. **数据模型是业务的真相。** 文档会撒谎、代码会绕路，schema 和核心表关系不会。
   进新项目先画 ER 草图，从 schema 反推业务规则（W1 建模训练的反向使用）。
2. **找「钱的路径」。** 公司靠什么收钱，那条链路就是理解优先级最高、最不能改坏的代码。
3. **带证据提问**：「我读了 X，预期 Y，实际看到 Z」——老人愿意答，自己也被迫先读。
4. **第一周搞清发布流程。** 部署管道、回滚方式、灰度机制是代码库的一部分；
   多数新人最后才学它，然后在第一次上线时付学费。
5. **前两周记三样东西**：术语表、谁知道什么、踩坑记录。这份笔记第三个月价值翻倍
   （每日笔记习惯的直接迁移）。

## 七、面试叙事：存量经验是资产，不是要藏的短板

- 「错误边界为什么还得用类组件」——真写过的人才答得细；
- 「HOC / render props 为什么被自定义 Hook 取代」——嵌套地狱亲历者讲得出具体痛点；
- 「接手类组件为主的存量项目怎么办」——只写过 hooks 的候选人答不好，你是优势方；
- 会写新栈的人多，**能在十万行存量代码里安全做手术的人少**。

一句话版本：

```text
大型存量系统多年（稳定性证据）
+ Next 14/15 现役经验（新范式证据）
+ 本仓库：带 CI、测试、鉴权、文档的现代全栈项目（正在进行的验证环节）
```

## 八、老技术清单：按「读得懂」和「写得了」分档

存量项目 90% 的场景是**读懂并安全修改**，不是从零用旧技术写新东西——学习目标是翻译能力。

**读得懂就够（需要时 1-2 天速览）**：类组件生命周期 ↔ useEffect 映射、经典 Redux
（store/action/reducer/connect + thunk/saga——国内电商 dva/umi 系存量大）、
webpack/babel 配置定位能力、moment.js 的 mutable 坑、PropTypes。

**必须真正掌握（因为不是旧技术）**：React Context 的角色边界（低频全局数据用它正合适，
高频状态会全树重渲染——它是现行 API，不是遗留）；Redux 思想（不可变更新、单向数据流
——库会过时，思想是读懂任何状态管理代码的底图）。

**不预学**：Vue 3（JD 明确要求再学，hooks 扎实后 Composition API 是同构概念，
按需 1-2 周量级；映射见 hooks 面试地图第四节）。
