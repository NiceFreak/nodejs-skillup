## 1. Node.js 运行时（事件循环）— W5 ★

### Q1. Node 是单线程吗？异步是怎么实现的？
**A:** 准确说是**执行 JavaScript 的是主线程**，不是"整个进程只有一个线程"。libuv 在底层提供事件循环、I/O 抽象和一个线程池。异步 I/O 大多交给操作系统的异步机制或 libuv 线程池，完成后把回调排进队列，**等主线程当前调用栈清空了再执行**。所以"异步"不等于"多开了 JS 线程"。
**◇ 边界**：线程池具体调度、`UV_THREADPOOL_SIZE` 的作用边界我正在补（W5 D2），目前能讲的是"它只影响部分线程池任务，不是让 JS 变多线程"。

### Q2. `process.nextTick`、Promise.then、setTimeout、setImmediate 的执行顺序？
**A:** 先分清三个互不隶属的容器：libuv 的**六个事件循环阶段**（timers → pending callbacks → idle/prepare → poll → check → close）、Node 的 **next tick queue**、V8 的 **microtask queue**。nextTick 和 Promise 都**不属于**那六个阶段。每次调度机会，Node 先清 next tick queue，再清 microtask queue，然后才推进事件循环阶段。setTimeout 的回调在 `timers` 阶段，setImmediate 在 `check` 阶段。
**★ 加分点（大多数人不知道）**：nextTick 和 Promise 谁先，**取决于模块系统**。我实测过同一份代码：
```
CommonJS：start → end → nextTick → promise
ESM：     start → end → promise → nextTick
```
因为 ESM 顶层求值本身就在一个 microtask checkpoint 里，求值期间注册的 Promise.then 会被并进当前正在清的 microtask 队列，nextTick 反而排在后面。所以"nextTick 永远最优先"是缺上下文限定的简化说法——它不会抢占当前调用栈，也不会中断已经开始的 microtask checkpoint。

### Q3. `setTimeout(fn, 0)` 是 0 毫秒后执行吗？和 setImmediate 谁先？
**A:** 不是。`0`（小于 1ms）会被归一化到约 1ms，它是**最早可调度的阈值**，不是精确执行时刻。顶层同时写 setTimeout(0) 和 setImmediate 时，**顺序不确定**——我跑 100 次是 94 次 setImmediate 先、6 次 setTimeout 先，受进程启动时机影响。只有在一个 I/O callback **内部**注册两者，才有稳定顺序（setImmediate 先，因为 poll 之后紧接着就是 check 阶段）。
**◇ 边界**：I/O callback 内部这个稳定顺序我记录里刚开头、还没跑完对照实验，是我明天要补的。

### Q4.（映射回业务）什么代码会阻塞事件循环？
**A:** CPU 密集的同步 JS 会。典型例子就在我自己的认证代码里：`bcrypt.compare` 是 CPU 工作，登录时错误密码要走完一次 compare（约 314ms），而邮箱不存在会提前 return（约 2ms）——这个可观察的耗时差，本质就是"主线程被 CPU 占用了多久"。数据库查询则是 I/O，不占主线程。区分"慢在 CPU 还是慢在 I/O"是我这周在补的判断力。

## 2. Express / 后端架构 — W2 ★（重点强项，尽量往这引）

### Q5. 讲讲你的项目分层。
**A:** 四层，单向依赖：**route**（URL 分发）→ **controller**（翻译官：读 req 参数、调 service、包 res）→ **service**（纯业务）→ **repository**（数据访问）。我有一把很简单的**检验尺子**：全局搜 `req`/`res`，它俩只允许出现在 controller；出现在 service 或 repository 就是越界了。
**★ 加分点**：单向依赖的根据是"**下层不该认识上层**"（依赖方向），跟 Express 是不是洋葱模型没关系——洋葱模型讲的是中间件横向串联，分层单向是纵向架构原则，换任何框架都成立。很多人会把这两个因果搭错。

### Q6. 分层的实际好处是什么？不会太啰嗦吗？
**A:** 最值钱的回报是**换数据库只动 repository**——service 是纯业务、不碰数据库细节，所以底层从写死数组换成 Mongoose 查询时，上面三层几乎不动。第二个是**可测试性**：service 不碰 req/res、不连真库，喂普通参数就能单独测，不用起 HTTP 服务器。至于"service 很薄像没用"——薄不等于该删，它是业务逻辑的预留位，今天没逻辑不代表明天不加"只返回激活用户""按权限过滤"。

### Q7. 400 和 404 有什么区别？在哪一层判断？
**A:** 400 是**请求本身格式错**（比如 `/users/abc`，id 不是数字），404 是**请求没错但资源不存在**（`/users/3`，格式合法查无此人）。落层上：repository 只返回 `undefined`（它不懂 HTTP），由 **controller** 把 falsy 翻译成 404；参数校验也放 controller，在碰 service 之前就拦掉非法输入。判断"有没有"在底层，"返回什么状态码"必须在 controller——还是那把"谁懂 HTTP"的尺子。

### Q8. `/users/:id` 里的 `:id` 是怎么变成参数的？
**A:** `:id` 是占位符，Express 在**路由匹配阶段**把 URL 对应位置的值抓出来、以冒号后的词为键名存进 `req.params`，所以是 `req.params.id` 不是 `req.id`，controller 只是读现成的值。而且值**永远是字符串**——URL 本质是文本，Express 不替你猜类型，这也是为什么底层比较数字 id 时要先 `parseInt`。顺带一组对照：路径参数 `req.params`、查询串 `req.query`、请求体 `req.body`。

## 3. 认证与鉴权 — W4 ★（强项，密码/JWT/RBAC/OAuth2 都能纵深）

### Q9. 密码怎么存？为什么用 bcrypt 而不是加密？
**A:** 存的是 `bcrypt.hash(password, 12)` 的哈希，不是明文也不是可逆加密。哈希是单向的——验证时用 `bcrypt.compare(明文, 哈希)`，而**不是**把输入重新哈希再比字符串（因为 bcrypt 每次带随机盐，同一密码两次哈希结果不同，字符串比对必然失败；compare 会从存的哈希里取出盐来算）。数据库里不出现明文，接口返回也剔掉 passwordHash。

### Q10. JWT 里放了什么？为什么不放角色？
**A:** payload 只放 `sub`（用户 id），不放 role/email/name。**不放 role 是刻意的**：如果 role 进了 token，那服务器端把用户提权/降权之后，旧 token 还带着旧角色，直到过期前都反映不了最新权限。把角色判断放到**每次请求实时查库**，才能保证"数据库当前状态"是唯一可信来源。token 用 `jwt.sign` 签、带 1 小时过期，密钥从环境变量来。
**★ 加分点**：这正好牵出 401 和 403 的区别。

### Q11. 401 和 403 的区别？
**A:** 401 是**认证**失败——"你是谁"没确认（没带 token、token 无效/过期），卡在 `validateToken` 中间件。403 是**授权**失败——身份确认了，但"你能不能"不通过（token 有效但角色不是 admin），卡在 `requireRole`。中间件我是分开接的：`validateToken → requireRole("admin") → controller`，认证在前、授权在后，`requireRole` 用 token 里的 `sub` 去查库拿当前 role 再判断。
**术语**：Authentication=认证=你是谁=401；Authorization=授权/鉴权=你能不能=403。中文"鉴权"口语常把两者混着说，我会刻意区分。

### Q12. 客户端能不能自己把自己提成 admin？
**A:** 不能。注册接口的入参解构只取 `{ name, email, password }`，请求体里就算塞 `role: "admin"` 也会被**静默忽略**——不是靠校验分支挡的，是源头根本没给它进来的机会（没有校验分支，就没有绕过校验的攻击面）。改角色只能走运维侧（我演示时用 mongosh），不是公开 API。

### Q13. 讲讲 OAuth2 授权码流程。
**A:** 六步：① 开发者先在第三方注册 App 拿 `client_id`/`client_secret`、登记 `redirect_uri`；② 用户点登录，浏览器带 `client_id`/`redirect_uri`/`state` 跳到授权服务器；③ 用户在第三方认证后，浏览器带一次性 `code` 重定向回我们的 callback；④ **后端**用 `code + client_secret` 换 `access token`（这步必须在后端，client_secret 绝不能进浏览器）；⑤ 后端用第三方 access token 请求资源服务器拿用户资料；⑥ 根据 provider+providerUserId 建/绑本地用户，签发**我们自己的 JWT**。
**★ 加分点（凭据边界）**：`code` 是一次性换票凭据、走浏览器；第三方 `access token` 只在后端、只用于访问第三方资源，**不等于**我们系统的登录凭据；本系统权限仍走自己的 JWT + 查库 role。`state` 防 callback 被串线（CSRF 式注入），`redirect_uri` 白名单防 code 被送到攻击者地址。
**◇ 边界**：这是**流程级**掌握，我没接真实第三方登录代码——这是我当时主动的范围取舍，不是没做完。

### Q14.（安全）你提到的计时枚举是什么？
**A:** 登录时我对"密码错"和"邮箱不存在"返回**完全一样**的 401 文案（反枚举，避免攻击者靠响应差异探测哪些邮箱注册过）。但**耗时**还有差异：密码错走完一次 bcrypt.compare（约 314ms），邮箱不存在提前 return（约 2ms），这个时间差仍可被计时攻击区分。我把它记为**已知安全遗留**，触发条件是上生产/公网，优先修复方案是补一次 dummy compare 拉平耗时 + 限流。我倾向于"讲清楚边界"而不是假装没有。

## 4. MongoDB / 数据 — W3（有真实深度，但广度是相对弱项）

### Q15. 你做过哪些聚合？
**A:** ★ 做过三个报表并竖切成真实接口：客户消费（`$match → $group → $sort`）、带客户名的 `$lookup` 关联、月度趋势（按 `$year`/`$month` 分组，自己从零设计）。聚合逻辑我是**分层放**的：像"最近 6 个月"这种业务规则由 service 持有、算出时间边界，那段 `aggregate([...])` 管道属于查询实现、放 repository。判据和分层同一套：问它是"数据库实现细节"还是"业务规则"。

### Q16. 查询慢怎么优化？★（这是你最能打的 DB 题）
**A:** 我不靠直觉，用 `explain` 的硬指标：`collectionScans`、`indexesUsed`、`totalDocsExamined`。举我做过的一个对照：一个关联 `name` 字段的 `$lookup`，建索引前是 `collectionScans:3`、`indexesUsed:[]`、扫 15 个文档；`createIndex({name:1})` 之后同一查询变成 `collectionScans:0`、`indexesUsed:["name_1"]`、扫 0 个文档。判断一个查询快不快，看这两个字段就够。核心原则其实一句话：**你查的字段有没有索引**——它贯穿 `$match` 和 `$lookup`，是一件事的两个应用。`_id` 永远自带唯一索引，所以按主键关联天然快。

### Q17. populate 和 $lookup？N+1 是什么？
**A:** populate 是 Mongoose 层的关联，底层可能变成多次查询，数据量大时有 N+1 风险（查 1 次列表 + 对每条再查关联）；`$lookup` 是数据库层一次聚合内完成关联。我做过两者的对比。
**◇ 待深入**：`$lookup` 的**子管道（sub-pipeline）**优化我还没吃透——现在是把整个关联文档搬进来再 `$project` 裁剪，我知道更省的写法是在关联阶段就用 pipeline 只取需要的字段，但"到底省多少"我想像做索引实验那样用 explain 的文档处理量对比出来，还没做。

### Q18.（诚实的广度题）事务、复制集、分片你了解吗？
**A:** ◇ 这些我目前是**概念级**了解，没有生产实战。我的实战深度集中在**单库的查询建模和索引优化**——聚合、explain、索引对照这些我能用数据说话。事务/复制/分片这类偏运维和规模化的部分，是我明确知道自己还没补到的边界；真需要时我会用同样的方法（先建立可测量的判据、再动手），而不是凭感觉。

## 6. 高频"为什么"速查（被连问时的一句话弹药）

| 问题 | 一句话 |
|---|---|
| 为什么验证密码用 compare 不用重新 hash 比对 | bcrypt 每次带随机盐，同一密码两次 hash 不同；compare 会从存的 hash 取盐来算 |
| 为什么 JWT payload 不放 role | 放了就无法反映提权/降权后的最新权限；角色应每次查库 |
| 401 vs 403 | 认证失败（你是谁）vs 授权失败（你能不能） |
| 为什么 req/res 只在 controller | 它是 HTTP↔业务的翻译层；下层不该懂 HTTP |
| 分层为什么单向 | 下层不该认识上层（依赖方向），与洋葱模型无关 |
| 换数据库要改几层 | 只改 repository，上面三层几乎不动 |
| 判断查询快慢看什么 | explain 的 collectionScans / indexesUsed / totalDocsExamined |
| nextTick 一定最先吗 | 不，ESM 顶层里 Promise 会先于 nextTick |
| setTimeout(0) 精确吗 | 不，0 是最早可调度阈值（约 1ms），非精确时刻 |
| 什么阻塞事件循环 | CPU 密集同步 JS（如 bcrypt），I/O 不阻塞主线程 |
