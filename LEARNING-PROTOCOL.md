# 学习状态与 AI 协作规范

本文件定义如何在每天开始、结束以及新开 AI 对话时恢复学习上下文。目标是让进度由仓库中的事实驱动，不依赖某一段聊天记录是否仍在上下文中。

本规范、状态文件、周计划、每日笔记和 AI 辅助记录统一使用中文书写。代码标识符、命令、协议名和无法自然翻译的技术术语可以保留英文。

---

## 1. 文件职责

| 文件 | 职责 | 更新时机 |
|---|---|---|
| `AGENTS.md` | AI 协作边界、黑白名单、辅助阶梯和 review 规则 | 协作原则发生变化时 |
| `README.md` | 整体路线、周目标和最终期限 | 总计划或周范围变化时 |
| `LEARNING-STATE.md` | **当前唯一状态入口**：做到哪、卡在哪、下一步是什么 | 每个阶段完成、计划改变、当天结束时 |
| `weekN-*/notes/weekN-plan.md` | 当周每日计划、验收标准和取舍门槛 | 周初建立，日终勾选 |
| `weekN-*/notes/dayN-*.md` | 当天完整学习过程、决策、错误和复盘 | 学习过程中滚动记录，日终收口 |
| 代码与测试 | 行为事实 | 实现和验证时 |
| Git diff/status | 尚未提交的实际变化 | 每次恢复状态时读取 |

`LEARNING-STATE.md` 不是日记，不复制长篇知识点；每日笔记不是当前任务看板，不承担跨对话入口职责。

---

## 2. 新对话固定读取顺序

新对话开始时，AI 只需先读取以下文件，不要扫描所有历史笔记：

1. `AGENTS.md`
2. `LEARNING-PROTOCOL.md`
3. `LEARNING-STATE.md`
4. 根 `README.md` 中当前周对应部分
5. `LEARNING-STATE.md` 指向的当周计划
6. `LEARNING-STATE.md` 指向的最近一篇每日笔记
7. `git status --short` 与当前任务相关文件的 diff
8. 当前步骤直接涉及的代码和测试

只有在当前状态文件明确引用、出现结论冲突或需要追溯设计理由时，才读取更早的笔记。

### 新对话启动口令

用户可以直接说：

> 按 `LEARNING-PROTOCOL.md` 恢复状态，告诉我当前进度、今天目标和第一步，不要先改代码。

AI 恢复后先用不超过 10 行说明：

- 当前周 / Day
- 已完成的最近验收点
- 当前阻塞或风险
- 今天唯一主线
- 立即开始的第一步

---

## 3. 每天开始流程

1. 读取状态文件、当周计划和最近日记。
2. 检查 git 状态，区分已知学习改动与意外改动。
3. 运行与当前主线直接相关的最小验证；若成本较高，先确认是否需要。
4. 把今天任务压缩成一个主线和至多一个附加项。
5. 明确当天止步条件：做到什么算完成，什么自动顺延。

开始新主题前，学习者应能回答：

```text
今天要解决什么问题？
最终可以演示或验证什么？
哪些内容今天明确不做？
```

---

## 4. 学习过程记录

每日笔记按以下结构滚动维护：

```text
标题 + 一段摘要
→ 今日目标 / 计划变化
→ 需求与概念
→ 自己的预测或设计
→ 实现过程
→ 错误现象与根因
→ 验证证据
→ 已完成 / 未完成
→ 明日入口
```

记录事实时区分：

- **事实**：代码、命令输出、测试结果已经验证。
- **推断**：根据 explain、日志或行为得出的解释。
- **待验证**：目前只有直觉或资料结论，尚未亲手证明。

AI 对黑名单知识点给出 L2（骨架）援助、或出现任何误给的 L3/L4 时，同步在 `DEBT.md` 记账（见 `AGENTS.md` 的「欠债记账」），并在日记额外记录：

```text
AI 辅助范围：哪些知识点 / 函数 / 分支
援助级别与场景：L2 骨架还是误给的 L3/L4，当时卡在哪
本人理解验证：复述 / 修改 / 测试完成了什么
延迟重建日期：何时按重建梯子脱离实现重建
```

## 5. 状态文件更新规则

发生以下任一事件时更新 `LEARNING-STATE.md`：

- 一个可验收阶段完成。
- 主线计划、范围或日期发生变化。
- 新增阻断问题。
- AI 对黑名单知识点给出 L2 援助（同步记入 `DEBT.md`）。
- 当天学习结束。

状态文件必须保持简短，并始终包含：

```text
当前周 / 日期 / Day
本周主题与硬截止时间
最近完成
当前主线
当前阻塞与风险
下一步（必须可直接执行）
验收命令或证据
需要读取的文件
AI 辅助记录与延迟重建
```

已解决的细节移入每日笔记，在状态文件中只保留一行结论和链接。

---

## 6. 每天结束流程

1. 运行与当天改动相称的测试或手动验证。
2. 按 `AGENTS.md` 的 review 规则判断是否有阻断性问题。
3. 收口每日笔记，明确哪些是事实、哪些未完成。
4. 更新当周计划的 checklist。
5. 更新 `LEARNING-STATE.md`，写出明天第一个动作。
6. 查看 git diff，确认没有误提交敏感信息、调试输出或实验索引清理遗漏。
7. 当天新增或修改了可演示 API 时，同步更新 Postman collection / `.http` 展示资产；这属于白名单收尾，不替代学习者负责的自动化测试设计。
8. 由学习者决定是否 commit；AI 不擅自提交。

---

## 7. 范围与优先级

优先级固定为：

```text
本周核心能力闭环
→ 会阻断闭环的缺陷
→ 关键路径测试与演示
→ 本周复盘
→ 上周遗留实验与优化
```

遗留项只有在本周主线达到当天止步条件后才能进入。出现截止时间压力时，优先砍独立扩展，不压缩核心链路的理解和验收。

---

## 8. 掌握验收

一个主题不以「代码跑通」单独判定掌握。至少满足：

1. 能不看代码画出或讲清调用链。
2. 能解释一个成功路径和两个失败路径。
3. 能预测修改一个需求会影响哪些层。
4. 能在延迟后脱离 AI 和文档重建核心链路。

若某知识点在 `DEBT.md` 有记录（曾获得 L2 援助、误给的 L3/L4 或重建卡档），第 4 项为最终验收的必选项。

## 9. 长期能力提醒（不增加每日负担）

本项目不只学习 Node.js、MongoDB、JWT 等技术表面，也借真实问题训练更长期的可迁移能力：

- **业务建模**：先把模糊需求变成明确契约、输入输出和范围。
- **正确性推理**：识别不变量、边界归属、状态变化和失败路径。
- **分层判断**：区分业务决策、数据访问、HTTP 表达和工具 API 各自的职责。
- **验证设计**：先预测结果，再用测试、日志或 explain 验证，而不是只看「跑起来了」。
- **技术表达**：能按「契约 → 推理 → 实现 → 证据 → 局限」讲清自己的方案。

这些是**长期观察维度，不是新增课程或每日 KPI**。执行时遵守：

1. 不单独安排一条「建模能力学习线」，不与当周技术主线竞争时间。
2. 只在真实任务自然暴露时，记录最多一个值得长期保留的思维模型。
3. 不要求每篇日记、每段代码都上升到抽象方法论；没有新洞察时不硬写。
4. 每周复盘最多挑一条反复出现、可迁移到其他技术的问题模式进行沉淀。
5. 语法、函数签名和库选项可以查询；只有业务模型、正确性逻辑或关键职责仍无法解释时，才视为掌握缺口。

### 三层区分

遇到「我不会写」时，先判断卡在哪一层：

| 层次 | 示例 | 掌握要求 |
|---|---|---|
| 业务模型 | 最近 6 个月是滚动窗口还是自然月 | 必须能独立澄清和定义 |
| 正确性模型 | 为什么相邻区间使用 `[start, end)` | 必须能解释并设计边界验证 |
| API 表达 | JavaScript `Date` 的具体方法名和参数 | 可以查文档、补全或使用白名单援助 |

不要因为第三层记忆不熟，就误判前两层没有掌握；也不要因为代码借助补全跑通，就默认前两层已经掌握。

### 工作与面试的共同证据

实际工作允许查 API，面试可能要求在有限时间内抽象表达。两者共同需要的能力证据是：

```text
先澄清契约
→ 给出具体例子
→ 说明不变量与边界
→ 决定职责和实现方向
→ 设计验证
→ 最后补齐 API 语法
```

这套顺序作为长期提醒即可，不纳入每日强制 checklist，也不因为某天没有体现而判定学习失败。
